# REVIEW — актуальное состояние ProfBot

Дата ревью: 2026-05-12  
Проверенная кодовая база: `/workspace/profbot` (текущая рабочая ветка)

## Итоговая оценка

Общая готовность к stage/MVP-запуску: **7.8 / 10**

Проект реализует основной пользовательский контур: регистрация, профиль, реферальная механика, daily-бонусы, магазин с AUTO/MANUAL доставкой, покупки с изменением баланса и админский контур (пользователи, товары, заказы, агрегированная статистика). Плюс в текущем состоянии есть рабочий поток Telegram Payments (инвойс → pre-checkout → зачисление баланса), что закрывает важный денежный сценарий на базовом уровне.

Ключевые ограничения до production: слабое тестовое покрытие по слоям роутеров/интеграций, Redis storage для FSM подключен, но эксплуатационное усиление всё ещё требуется, отсутствие hardening по конкурентным списаниям под высокой нагрузкой и отсутствие полноценной эксплуатационной обвязки (метрики/алерты/трейсинг).

## 2) Что фактически реализовано (по коду)

### 2.1 Инициализация приложения

- Приложение стартует через `main.py`: валидирует runtime-конфиг, поднимает таблицы через `Base.metadata.create_all`, регистрирует middleware и роутеры, затем запускает polling.  
- Подключены пользовательские и админские роутеры, включая магазин, бонусы, профиль, статистику и платежи.
- FSM storage переведен на Redis (`RedisStorage`) вместо in-memory, с закрытием Redis-клиента при завершении polling.

### 2.2 Конфигурация

- Конфиг на `pydantic-settings`, `.env` поддерживается.
- Проверяются обязательные переменные: `BOT_TOKEN`, `DATABASE_URL`, `PAYMENT_TOKEN`.
- Поддержаны поля для PostgreSQL/Redis и список `ADMINS`.

### 2.3 Магазин и покупки

- `/shop` и кнопка `🛒 Магазин` выводят товары с inline-пагинацией.
- При покупке:
  - списание баланса и фиксация транзакции происходят через сервис/репозиторий,
  - для `AUTO` выдается `delivery_content` (либо fallback-сообщение, если контент пуст),
  - для `MANUAL` создается заказ, пользователю отправляется подтверждение, админам отправляется уведомление с деталями заказа.

### 2.4 Платежи (пополнение)

- По кнопке `💰 Пополнить` выводятся тарифы (preset-пакеты кристаллов в UZS).
- По выбору тарифа бот отправляет Telegram invoice с `provider_token`.
- `pre_checkout_query` подтверждается автоматически.
- Добавлены проверки payload/суммы/владельца платежа на этапах pre-checkout и successful_payment.
- Добавлены retry-попытки зачисления баланса (до 3 попыток) с логированием ошибок и пользовательским fallback-сообщением при исчерпании попыток.

### 2.5 Админский контур

- `/admin` открывает админ-меню.
- Реализована `📊 Общая статистика` и `/admin_stats`:
  - пользователи (count, суммарный баланс),
  - товары (total/AUTO/MANUAL),
  - заказы (total/open),
  - экономика (количество транзакций и оборот как сумма абсолютных значений).

### 2.6 Тесты

- Есть набор unit-тестов репозиториев на in-memory SQLite:
  - создание пользователя с бонусом регистрации и статистикой,
  - запрет ухода баланса в минус,
  - базовые операции статистики,
  - создание manual-продукта,
  - lifecycle заказа (open → closed).
- Тестов роутеров, сервисов верхнего уровня, платежей и e2e-сценариев нет.

## 3) Сильные стороны текущей реализации

1. **Хорошее слоистое разделение** (routers/services/repositories/models), что упрощает развитие.
2. **Основной пользовательский value-path рабочий**: пользователь может прийти, получить валюту, купить товар, получить auto-выдачу или manual-обработку.
3. **Админская операционная видимость улучшена** благодаря общей статистике и ручным заказам.
4. **Платежный контур уже не фиктивный**: Telegram invoice flow реально интегрирован.

## 4) Риски, долги и ограничения

### P1 (высокий приоритет)

1. **Конкурентный доступ к балансу**: для high-load сценариев нужны более строгие гарантии атомарности (особенно при масштабировании и PostgreSQL-конкуренции).
2. **Покрытие тестами критичных сценариев**: отсутствуют интеграционные тесты покупок/платежей/админских действий и регрессионные тесты роутеров.
3. **Надежность платежного контура**: retry и базовые проверки добавлены, но всё ещё нет выделенной модели платежей/идемпотентности/аудита статусов (успех, повтор, откат, reconciliation).

### P2 (средний приоритет)

1. Нет rate limiting / антифлуда.
2. Observability ограничена логами; нет метрик, алертов, централизованного трейсинга.
3. `create_all` на старте полезен для dev/stage, но для production лучше управлять схемой через строгий миграционный процесс.

## 6) Рекомендованный план развития

### Ближайшие 1–2 итерации

1. Добавить интеграционные тесты для:
   - покупок `AUTO`/`MANUAL`,
   - уведомлений админам,
   - платежного сценария (invoice/payload/successful_payment).
2. Усилить консистентность экономики:
   - проверить/добавить транзакционную защиту и idempotency для денежных операций.
3. Перевести FSM на Redis storage (уже подготовлена инфраструктурная база в конфиге/compose).
4. Ввести минимальный anti-spam/rate limit на чувствительных хендлерах.

### Среднесрочно

1. Выделить полноценную платежную подсистему (`payments` таблица/статусы/аудит/reconciliation).
2. Добавить эксплуатационные SLO/метрики/алерты.
3. Формализовать release checklist (миграции, smoke тесты, rollback-план).
