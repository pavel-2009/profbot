## Что закрыто в последнем цикле

### 1) AUTO-доставка

- При успешной покупке товара с `delivery_type=AUTO` пользователь теперь получает отдельное сообщение с `product.delivery_content`.
- Если контент не заполнен, отправляется fallback-сообщение для прозрачного UX.

### 2) Уведомления админам по MANUAL-заказам

- При покупке manual-товара пользователю показывается подтверждение ожидания доставки.
- Всем `ADMINS` отправляется уведомление через `bot.send_message(...)` с идентификатором пользователя, товаром и подсказкой проверить `/orders`.

### 3) Админ-меню и агрегированная статистика

- Удалена лишняя кнопка `💰 Управление балансами` из reply-меню админа.
- Добавлен экран `📊 Общая статистика` (и команда `/admin_stats`) с агрегатами:
  - пользователи (count, суммарный баланс),
  - магазин (count товаров, AUTO/MANUAL),
  - заказы (всего/открытые),
  - экономика (число транзакций, оборот по модулю суммы).

## Фактическая функциональность на текущий момент

### Пользовательские сценарии

- `/start` создаёт пользователя и статистику.
- Реферальные бонусы работают (новому пользователю и пригласившему).
- `/shop` показывает товары с пагинацией.
- Покупка списывает баланс и записывает транзакцию.
- AUTO-товары выдают контент сразу после покупки.
- MANUAL-товары создают заказ для админа.
- `/daily`, `/rating`, `/stats`, профиль и история транзакций работают базово.

### Админские сценарии

- `/admin` открывает админ-меню.
- `/users` — список/карточка пользователя/история транзакций/зачисление баланса.
- `🛒 Управление товарами` — добавление и редактирование товара.
- `/orders` — просмотр открытых заказов и закрытие.
- `📊 Общая статистика` / `/admin_stats` — агрегированная аналитика по проекту.

## Технические риски и ограничения

### P1 (высокий приоритет)

1. **Платежи не реализованы**: кнопка `💰 Пополнить` есть, полноценной интеграции оплаты нет.
2. **Миграции для существующих БД**: текущая base-миграция ориентирована на новую БД; для апгрейда существующих стендов нужна отдельная incremental migration.
3. **Конкурентность баланса**: текущая модель достаточна для MVP, но для production нужна усиленная атомарность (особенно после переезда на PostgreSQL).

### P2 (средний приоритет)

1. Нет handler/integration/e2e тестов, нет тестов конкурентной покупки.
2. Redis поднят как заготовка, но FSM всё ещё в памяти; rate limiting отсутствует.
3. Логирование базовое: нет структурированных JSON-логов и корреляции запросов.
4. UX длинных списков в админке может упираться в лимиты Telegram-сообщений.
