# REVIEW — актуальное состояние ProfBot

Дата ревью: 2026-05-12  
Проверенная кодовая база: текущее состояние рабочей директории `d:\ITWork\profbot`

## Итоговая оценка

Общая готовность к небольшому stage/MVP-запуску: **6.8 / 10**

Проект больше не находится в состоянии, где базовые сценарии заказов/manual-доставки и статистики заведомо падают из-за несоответствия ORM и кода. Исправлены основные P0-дефекты: модель заказов, создание manual-заказа, enum-статусы, статистика транзакций, Alembic-схема, защита callback-админки, источник daily bonus, удаление `execute_with_retry`, проверка обязательных настроек и минимальные unit-тесты репозиториев.

До production всё ещё не хватает платежей, полноценной выдачи auto/manual-товаров, нормальной миграционной стратегии для уже существующих БД, rate limiting, мониторинга и более широких тестов. Но как рабочий MVP-каркас проект стал заметно ровнее.

## Что изменено в последнем проходе

### ORM и заказы

- В `Order` убран конфликт с несуществующим `quantity`.
- `ordered_at` получил дефолт `datetime.utcnow`.
- Статусы приведены к единому enum `OrderStatus.OPEN / OrderStatus.CLOSED`.
- `OrderRepository.create_order()` создаёт заказ без `quantity` и не делает лишний commit внутри создания.
- `OrderRepository.complete_order()` переводит заказ в `CLOSED` и задаёт `completed_at`.
- Админский вывод заказов больше не обращается к `order.quantity`, показывает `Status`.

### Manual-доставка

- `ShopService.buy_product()` сравнивает `product.delivery_type` с `DeliveryType.MANUAL`.
- При покупке manual-товара заказ создаётся через `OrderRepository.create_order()`.
- Уведомления админам при покупке manual-товара временно отключены по текущему решению.
- Пользователь получает сообщение, что покупка оформлена и ожидает доставки от администратора.

### Статистика и транзакции

- В callback полной истории транзакций используется `tx.reason`, а не несуществующий `tx.description`.
- В `/stats` расчёт самой частой покупки использует `Transaction.reason`.
- В `Statistics` добавлено поле `active_invited_users`, которое уже использовалось в репозитории и схемах.

### Daily bonus

- Поля `daily_bonus_last` и `daily_bonus_streak` удалены из `User`.
- Источник состояния daily bonus теперь только `Statistics.last_bonus` и `Statistics.last_activity_track_start`.

### DB/retry

- `execute_with_retry` полностью удалён из `bot/core/db.py`.
- Все использования `execute_with_retry(...)` заменены на обычный `session.flush()`.
- Это соответствует плану дальнейшего перехода на PostgreSQL вместо усложнения SQLite-ретраев.

### Конфиг и логирование

- Добавлен `validate_runtime_config()`.
- На старте проверяются `BOT_TOKEN` и `DATABASE_URL`.
- Логирование вынесено в `bot/core/logging.py`.
- `main.py` использует централизованный `setup_logging()`.

### Админские роутеры

- `admin/users.py` и `admin/orders.py` теперь защищают не только `message`, но и `callback_query`.
- В нескольких местах исправлен паттерн использования сервиса после выхода из `async with session`.

### Alembic

Базовая миграция `alembic/versions/0dda3e368ddf_.py` приведена ближе к текущим моделям:

- `orders.status` теперь enum `OPEN/CLOSED`;
- из `users` убраны `daily_bonus_last` и `daily_bonus_streak`;
- `statistics.last_bonus` теперь `DateTime`;
- добавлено `statistics.active_invited_users`.

Важно: это обновление базовой миграции хорошо для новой пустой БД. Если уже есть production/stage БД со старой схемой, нужна отдельная incremental migration, а не только правка initial migration.

### Тесты

Добавлен файл `tests/test_repositories.py` с минимальными unit-тестами репозиториев:

- создание пользователя, статистики и регистрационного бонуса;
- отказ транзакции при отрицательном балансе;
- обновление статистики;
- создание manual-товара;
- создание и закрытие заказа.

- `main.py` — точка входа, проверка runtime-конфига, настройка логирования, создание БД через `Base.metadata.create_all`, подключение роутеров.

- `bot/models` — модели `User`, `Statistics`, `Transaction`, `Product`, `Order`.

- `bot/routers/admin` — админские сценарии.
- `bot/middlewares` — статистика и проверка администратора.
- `tests` — минимальные unit-тесты репозиториев.

## Что реализовано и выглядит рабочим

### Регистрация и рефералка

- `/start` создаёт пользователя.
- Поддерживается реферальный код из аргумента `/start`.
- Есть проверка невалидного реферального кода.
- Есть защита от self-referral.
- При регистрации начисляется `+100`.
- При регистрации по рефералке новому пользователю и пригласившему начисляется по `+50`.
- Профиль и реферальный экран показывают базовую статистику приглашений.

### Баланс и транзакции

- `apply_balance_transaction()` меняет баланс и пишет запись в `transactions`.
- Баланс не может уйти в минус.
- Статистика транзакций и трат обновляется.
- Для SQLite это всё ещё read-modify-write без жёсткой защиты от гонок, но для минимального MVP приемлемо.

### Ежедневный бонус

- `/daily` и кнопка `🎁 Бонус` начисляют ежедневный бонус.
- Состояние бонуса хранится в `Statistics`.
- Повторное получение ограничено интервалом 24 часа.
- При длинном перерыве обновляется `last_activity_track_start`.

### Магазин и заказы

- `/shop` показывает активные товары с пагинацией.
- Покупка списывает баланс через общую транзакционную операцию.
- Manual-товар создаёт заказ.
- Auto-товар пока не выдаёт `delivery_content`, это ещё не реализовано.
- Админ может просматривать открытые заказы и закрывать их.

### Админ-панель

- `/admin` показывает меню.
- Админские message и callback handlers в основных админ-роутерах защищены middleware.
- Есть просмотр пользователей.
- Есть карточка пользователя.
- Есть админское пополнение баланса.
- Есть история транзакций пользователя.
- Есть FSM добавления товара.
- Есть FSM редактирования названия, описания и цены товара.
- Есть базовое управление заказами.

### Статистика

- `StatisticsMiddleware` обновляет активность в фоновой задаче.
- `/rating` показывает топ по балансу и место пользователя.
- `/stats` показывает активность, экономику, рефералы и покупки.
- `stats_graph_30d` строит текстовый график изменения баланса за 30 дней.

## Оставшиеся важные проблемы

### 1. Нет полноценной выдачи товара

У `Product` есть `delivery_content`, но:

- админская FSM добавления товара не запрашивает содержимое auto-доставки;
- покупка auto-товара не отправляет пользователю контент;
- manual-доставка создаёт заказ, но полноценного процесса выдачи/комментария/уведомлений пока нет.

Приоритет: **P1**.

### 2. Платежи не реализованы

В меню есть кнопка `💰 Пополнить`, но пользовательского сценария пополнения баланса нет. Yookassa или другая платёжная интеграция отсутствует.

Приоритет: **P1**.

### 3. Миграции подходят только для новой базы

Initial Alembic migration обновлена под текущие модели. Но если есть уже существующая база со старой схемой, ей нужна отдельная миграция изменения схемы:

- удалить старые `users.daily_bonus_*`;
- изменить `statistics.last_bonus`;
- добавить `statistics.active_invited_users`;
- привести `orders.status`.

Приоритет: **P1** для stage/prod с существующей БД.

### 4. Баланс не защищён от высокой конкуренции

SQLite и текущий ORM read-modify-write достаточны только для малой нагрузки. После перехода на PostgreSQL стоит сделать:

- условный update баланса;
- транзакционные блокировки;
- идемпотентные операции;
- тест на конкурентные покупки.

Приоритет: **P1/P2**.

### 5. Тестов всё ещё мало

Есть только минимальные unit-тесты репозиториев. Нет handler-тестов, интеграционных тестов, миграционных тестов, тестов конкуренции и тестов FSM.

Приоритет: **P2**.

### 6. Redis пока не используется

Redis поднимается в `docker-compose` и есть `bot/core/redis.py`, но:

- FSM хранится в `MemoryStorage`;
- rate limiting отсутствует;
- кеширование отсутствует.

Приоритет: **P2**.

### 7. Логирование централизовано, но ещё базовое

`setup_logging()` вынесен отдельно, но пока это обычный текстовый logging. Для production нужны JSON-логи, correlation id/telegram id в контексте и нормальная ротация.

Приоритет: **P2**.

## Замечания по UX

- Help показывает не все команды: нет `/daily`, `/rating`, `/stats`, `/main`, `/admin`, `/orders`, `/users`.
- При auto-покупке пользователь получает только alert об успехе, без полезного результата.
- При manual-покупке уведомление админам отключено, поэтому админ должен сам проверять раздел заказов.
- В админке длинные списки пользователей/заказов могут стать слишком большими для одного сообщения.

## Замечания по безопасности

- Админские роутеры стали защищены лучше: callback middleware добавлен в `admin/users.py` и `admin/orders.py`.
- `BOT_TOKEN` проверяется перед запуском.
- Нет rate limiting.
- Нет anti-spam.
- Нет разделения ролей внутри админки.
- Нет полноценного аудита админских действий, кроме транзакций пополнения баланса.
