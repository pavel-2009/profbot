# REVIEW.md — актуальное ревью проекта ProfBot

## 1) Краткий вывод

Проект находится в состоянии **раннего MVP**: базовые пользовательские сценарии `/start`, `/profile` и витрина `/shop` реализованы, данные хранятся в БД, есть модели пользователей/товаров/транзакций/статистики.

Однако текущая кодовая база содержит **несколько критичных рассинхронов между слоями model/repository/service**, из-за которых часть бизнес-логики (в первую очередь операции с транзакциями при покупке) в реальном рантайме будет падать.

**Оценка готовности к MVP: ~55%.**

---

## 2) Что реально есть в проекте на 2026-04-29

### 2.1 Реализованные пользовательские команды

- `/start`:
  - проверка существования пользователя;
  - регистрация нового пользователя;
  - поддержка реферального аргумента `start=<code>`;
  - защита от невалидного и самореферального кода;
  - приветственный ответ и выдача главного меню.

- `/profile`:
  - загрузка пользователя и агрегированного профиля;
  - вывод баланса, базовой статистики, реферальной ссылки;
  - вывод последних транзакций (до 5 записей в тексте ответа).

- `/shop`:
  - вывод активных товаров из БД с пагинацией;
  - перелистывание страниц через callback;
  - кнопки выбора товара (сейчас обработчик покупки — заглушка с alert).

### 2.2 Что есть на уровне доменной модели

- `users` — пользователь и баланс.
- `products` — товары магазина, включая `delivery_type`, `delivery_content`, `is_active`.
- `transactions` — отдельная таблица транзакций существует.
- `statistics` — отдельная таблица статистики пользователя существует.

### 2.3 Инфраструктура

- Инициализация схемы БД при старте через `Base.metadata.create_all`.
- Асинхронная SQLAlchemy-сессия.
- Redis используется для генерации/резерва реферальных кодов.

## 4) Ключевые технические находки

### 4.1 Сильные стороны

1. **Чёткая слоистость:** routers → services → repositories.
2. **Асинхронный стек** (aiogram + SQLAlchemy async).
3. **Pydantic-контракты профиля** (`UserProfileSchema`, `UserStatsSchema`, `TransactionSchema`).
4. **Базовая обработка ошибок** в роутерах и логирование критичных веток.

### 4.2 Критичные проблемы (P0)

1. **Несовместимость `TransactionRepository.add_transaction` и `Transaction` model.**
   - В модели поля: `balance_after`, `reason`.
   - В репозитории создаётся `Transaction(..., description=...)` без `balance_after`.
   - Это приведёт к ошибке создания транзакции.

2. **`/shop` не вызывает реальную покупку.**
   - Callback `shop_buy` сейчас только показывает alert, без списания баланса, без выдачи товара, без записи транзакции.

3. **Статистика не инициализируется при регистрации.**
   - Таблица `statistics` используется в профиле, но создание записи статистики для нового пользователя не гарантировано.
   - В результате профиль часто будет показывать нули через fallback, а не реальные данные.

### 4.3 Средние проблемы (P1)

1. **Реферальная статистика не обновляется** (приглашённые/заработано по рефералке).
2. **Непоследовательность нейминга в транзакциях** (`reason` vs `description`).
3. **Есть неиспользуемые импорты и технический шум** (например, лишние импорты в ряде репозиториев).
4. **Нет миграционного процесса в runtime** (alembic присутствует, но фактически БД поднимается через `create_all`).

### 4.4 Низкий приоритет (P2)

1. Нет автотестов на критичные сценарии (`/start`, `/profile`, `/shop`).
2. Нет явной бизнес-валидации отрицательных кейсов покупок в user-facing ответах.
3. Не хватает операционной наблюдаемости (метрики/структурированные события).

## 5) Рекомендуемый план работ

### P0 (сразу)

1. Привести контракт транзакций к единому виду:
   - либо `reason`, либо `description`, но единообразно во всех слоях;
   - явно передавать/считать `balance_after` при каждой операции.
2. Сделать рабочий `shop_buy`:
   - вызов `ShopService.buy_product`;
   - корректные сообщения об успехе/ошибке;
   - запись транзакции + обновление статистики в одной транзакции БД.
3. Инициализировать `statistics` при регистрации пользователя.

### P1

4. Дополнить реферальный сценарий обновлением статистики (`invited_users`, `earned_crystals_via_referrals`).
5. Реализовать `/daily` с 24h cooldown и streak-логикой по ТЗ.
6. Добавить smoke/integration тесты минимум на `/start` и покупку в `/shop`.

### P2

7. Завести админ-контур (начисления/списания/товары).
8. Перевести жизненный цикл схемы БД на миграции Alembic как основной путь.

---

## 6) Финальный статус

Сейчас ProfBot — это **рабочий прототип**, который уже покрывает основной пользовательский путь входа и просмотра профиля, а также отображение каталога товаров. Для перехода к полноценному MVP нужно закрыть критичные дефекты в транзакционном контуре и довести покупку в магазине до конца бизнес-цепочки.
